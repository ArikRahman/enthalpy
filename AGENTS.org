* Project Overview
Enthalpy is a ClojureScript port and rework of the ColemakCamp typing trainer
(https://github.com/ColemakCamp/colemakcamp.github.io), reoriented around the
Enthium v14 keyboard layout. The primary feature is an interactive typing trainer
with progressive level unlocking, a live cheatsheet keyboard, and WPM/accuracy
scoring. Secondary features are supplementary documentation pages (rationale,
stats, comparisons, learning guide) reachable via hash routes.

The original ColemakCamp source (JavaScript) lives in the repository root and
serves as the reference implementation for UI behaviour and word-list logic.
The ClojureScript rewrite lives in ``src/`` and compiles to ``docs/``.

** What ColemakCamp provides (reference only — do not ship as-is)
- Typing trainer loop: prompt words → capture input → score hits/errors → next
- Progressive level system: each level unlocks more letters (lvl1–lvl8)
- Cheatsheet keyboard: visual key map shown alongside the typing area
- Settings: capital letters, punctuation, backspace correction, time/word limits,
  sound on keystroke/error
- Layout selector: Colemak, Colemak-DH, Tarmak, QWERTY, custom, etc.
- Word list: filtered by the letters available at the current level

** What Enthalpy changes
- Single layout: Enthium v14 only. The layout selector is removed.
- Level letter sets: redefined in Enthium frequency order (highest-freq letters
  first), not Colemak order.
- Branding, copy, and colours: Enthium/Enthalpy identity.
- Stack: ClojureScript + Reagent instead of vanilla JS. All state in Reagent atoms.
- Supplementary docs: rationale, stats, compare, and learning-guide pages added
  as hash routes alongside the trainer.

* Typing Trainer Architecture
All trainer state lives in a single Reagent atom in ``trainer.cljs``.

** State shape
#+BEGIN_SRC clojure
(def state
  (r/atom
    {:level           1        ; 1–8
     :mode            :words   ; :words | :time
     :limit           15       ; word count or seconds depending on mode
     :words           []       ; current prompt word list
     :word-index      0        ; index into :words of the active word
     :input           ""       ; current text-field value
     :input-error?    false    ; true when current input doesn't match target prefix
     :hits            0        ; correctly completed words
     :errors          0        ; incorrect keystrokes
     :started?        false    ; true once first keystroke received
     :finished?       false    ; true when limit reached
     :start-ms        nil      ; js/Date.now() at first keystroke
     :elapsed-ms      0        ; updated by RAF loop while running
     :raf-id          nil      ; requestAnimationFrame handle for the timer
     :show-cheat?     true     ; show cheatsheet keyboard
     :sound-stroke?   false    ; play sound on keystroke (wired to UI toggle; audio NYI)
     :sound-error?    false    ; play sound on error (wired to UI toggle; audio NYI)
     :allow-caps?     false    ; include capital letters in prompts
     :allow-punct?    false    ; include punctuation in prompts
     :backspace-req?  false    ; require backspace to correct errors
     :emulate-qwerty? false})) ; translate physical US QWERTY keys to Enthium positions
#+END_SRC

** Level definitions
Defined in ``words.cljs`` as ``level-new-letters`` — a map from level number to
the set of letters newly introduced at that level. Letter order follows Enthium
v14 home-row frequency (highest first), not Colemak order.

#+BEGIN_SRC clojure
(def level-new-letters
  {1 #{"h" "t" "n" "s" "e" "a" "i" "r"}
   2 #{"o" "c" "l" "d"}
   3 #{"u" "y" "m" "p"}
   4 #{"b" "w" "f" "g"}
   5 #{"k" "v" "j" "x"}
   6 #{"'" "," "." ";"}
   7 nil   ; all words — no restriction
   8 nil}) ; sentence mode
#+END_SRC

Each level's available set is the cumulative union of all letter sets from
level 1 up to and including the current level, computed by ``level-letters`` in
``words.cljs``. The word list is filtered to only words constructable from
that set. Levels 7 and 8 are unrestricted; level 8 uses the sentence corpus
instead of the word list.

** Trainer loop
1. ``init-round!`` — pick N words from the filtered list, reset indices/input.
2. User types in ``#userInput`` (a plain ``<input>``).
3. On each ``on-change``: compare current token against ``(nth words word-index)``.
   - Trailing space = word submitted. If correct → increment hits, advance index.
   - Wrong character → increment errors (optionally play error sound).
4. On word-limit or time-limit reached → set ``:finished? true``, compute WPM.
5. ``reset-round!`` resets state and calls ``init-round!``.

WPM formula (standard): ``(/ hits (/ elapsed-ms 60000))``.

** QWERTY emulation layer
Implemented in ``src/enthalpy/emulation.cljs``. When ``:emulate-qwerty? true``
the trainer intercepts every ``keydown`` event, translates the physical US QWERTY
key to the Enthium v14 character at the same physical position, and drives
``on-input-change!`` directly. The ``<input>`` value always shows Enthium
characters; QWERTY translation is transparent to the rest of the trainer.

The toggle button is labelled **US KB** in the settings row. Toggling it does
not reset the current round.

Key API in ``emulation.cljs``:
- ``qwerty->enthium`` — map of 31 physical QWERTY keys → Enthium characters
- ``enthium->qwerty`` — reverse map (for future hint rendering)
- ``translate-key [ch allow-caps?]`` — translate a single character
- ``translate-string [s allow-caps?]`` — translate a whole string

CapsLock is excluded from the map because browsers cannot reliably intercept it
at the application level.

** Cheatsheet keyboard
Rendered by ``layout/keyboard-diagram`` (already implemented). Shown below the
typing area when ``:show-cheat? true``. Toggled by a button in the toolbar.
Highlights the current target key based on the next character to type (future
enhancement — not required for v1).

* Version Control
This repository uses **Jujutsu** (``jj``) in colocated mode — the ``.jj/``
directory sits alongside ``.git/`` so both tools stay in sync. GitHub pushes
use ``jj git push``. Never run raw ``git commit``; use the ``just`` recipes
below instead.

** Core concepts
| jj concept    | git equivalent         | Notes                                              |
|---------------+------------------------+----------------------------------------------------|
| Change (@)    | Working tree + staging | Always exists; automatically tracks all edits      |
| Bookmark      | Branch                 | ``main`` bookmark tracks the ``main`` git branch   |
| ``jj commit`` | ``git commit``         | Seals @ with a message and opens a new empty child |
| ``jj describe``| ``git commit --amend`` | Rewrites the message of an existing change         |
| ``jj diff``   | ``git diff``           | Shows working copy changes                         |
| ``jj log``    | ``git log``            | Visualises the change graph                        |

** Typical workflow
#+BEGIN_SRC sh
# 1. Edit files — jj tracks all changes automatically, no staging needed.

# 2. Check what changed
just st          # jj status — files changed in @
just diff        # full diff of working copy

# 3. Commit and push
just snap "Add feature X"
# expands to:
#   jj commit -m "Add feature X"   ← seals @ with message, opens new empty @
#   jj bookmark set main --revision @-   ← advances main bookmark
#   jj git push --bookmark main    ← pushes to GitHub

# 4. Or run CI checks first, then commit + push
just ship "Add feature X"
# expands to: just ci && just snap "Add feature X"
#+END_SRC

** Bookmark / branch model
- ``main`` is the only long-lived bookmark.
- After ``jj commit``, ``@`` is a new empty child and ``@-`` holds the sealed
  change. ``jj bookmark set main --revision @-`` moves ``main`` to that commit
  so ``jj git push --bookmark main`` sends it to GitHub.
- Do not create additional bookmarks unless you are deliberately working on a
  feature branch.

** Recovering from mistakes
#+BEGIN_SRC sh
jj diff                   # inspect before acting
jj restore --from @- .    # discard all working copy changes (dangerous!)
jj abandon @              # discard current empty change (just abandon recipe)
jj undo                   # undo the last jj operation (safe, always reversible)
jj op log                 # audit full operation history
#+END_SRC

** Colocated repo rules
- ``.jj/`` is in ``.gitignore`` — it is never committed to git.
- Never run ``git commit`` directly; the colocated mode keeps both in sync only
  when you go through ``jj``.
- ``jj git fetch`` + ``jj rebase -d main@origin`` is the pull/rebase workflow.
- ``git status`` still works for quick inspection but defer to ``just st``.

** Syntax check
Run ``just check`` from ``enthalpy/``. It invokes ``scripts/check-syntax.sh``
which parses every ``.cljs`` file individually and reports OK / ERR per file.
All files must print OK before committing.

** Full test suite
Run ``just test`` from ``enthalpy/``. It invokes ``scripts/test.clj`` via
``clj -M`` and runs 11 integrity suites covering syntax, section registry,
trainer state shape, emulation map coverage, and the build artifact.

** Stale server
If ``just dev`` prints ``already started``, stop the background JVM first:

#+BEGIN_SRC sh
just stop   # graceful shutdown
just kill   # force-kill if stop hangs
#+END_SRC

* Verified Spec-Driven Development (VSDD)

Use VSDD for all non-trivial changes in Enthalpy: the specification is updated
first, implementation follows, and verification decides whether the work is
done. In this repository, ``AGENTS.org`` is part of the operative spec, not just
a README [web:6].

** VSDD process for this repo
1. Identify the spec surface affected by the task.
   - Routing change -> update the Routes section.
   - Trainer behaviour change -> update Typing Trainer Architecture or state
     shape.
   - New doc section type -> update Documentation Sections or Hiccup Authoring
     Reference.
   - Styling convention change -> update Styling rules or CSS token guidance.
2. Edit the spec first when behaviour, structure, or workflow changes.
3. Implement only what the updated spec requires.
4. Verify against the spec, not against intuition:
   - ``just check`` prints OK for all ``.cljs`` files.
   - ``just test`` exits 0 with all suites green.
   - ``just build`` completes with 0 warnings.
   - Affected route or feature behaves as described in the spec.
   - Mobile layout still works at 768px when relevant.
5. If implementation and spec disagree, resolve the mismatch explicitly. Do not
   leave silent drift between code and ``AGENTS.org``.

** Definition of done under VSDD
A change is done only when all of the following are true:
- The relevant spec in ``AGENTS.org`` matches the shipped behaviour.
- The code compiles and the required verification steps pass.
- No forbidden patterns were introduced, including inline styles, hand-edited
  compiled assets, HTML entities in strings, or non-Bun workflows.
- The change is minimal, coherent, and reversible.

** Practical rule
If you are about to change behaviour and have not updated the corresponding
specification section, you are probably moving too fast. Stop, patch the spec,
then continue with implementation.
